## 1. 総括 ― このカンファレンスで得た3つの学び

報告者として、今回のカンファレンスの本質的な学びは次の3点だと考えています。新機能の数より、この3点が自社にとって重要です。

1. **成果を決めるのはモデルの賢さではなく「評価・ワークフロー・組織設計」**
   楽天・みずほ・NRI が口を揃えて言っていた最重要メッセージ。同じモデルでも、業務をどう設計し、どう評価し、組織でどう運用するかで成果が何倍も変わる。逆に言えば、ここはモデルに依存せず**自社の努力で差がつく領域**。

2. **開発の仕事が「自分でやる」から「舵を取る（steering）」へ移りつつある**
   Claude Code の新機能群（自動レビュー・並列実行・定期自動実行・自動記憶）は、いずれも「人が手を動かす量を減らし、人は方向づけと例外対応に集中する」方向を向いている。

3. **価値の源泉は「課題を言語化する力」と「ドメイン知識」に移った**
   ソフトを作る技術的な障壁はほぼ消えつつある。これからは業務を深く理解し、何を作るべきかを定義できる人（エンジニアに限らない）が強い。だからこそ全社展開と、自社固有の知識をツールに注入する設計が効いてくる。

## 2. 主要な学びと自社への示唆

### 学び① モデルは「次世代前提」で設計したほうが得（Fable 5 発表）
- 本日 **Claude Fable 5**（第5世代モデル）が発表され、Claude Code 全ユーザーが利用可能に。長時間の自律作業・100万トークンの処理・指示前の先回り行動が強化された。安全対策を強めた上位版が **Mythos 5**。
- 登壇者（Anthropic）の主張：**「今のモデルではなく、次に出るより賢いモデルを前提に作れ」**。モデルが賢くなるほど、複雑な作り込みより、ファイルやサンドボックスといった単純な道具を渡すだけで動くようになる。
- **自社への示唆**：作り込みすぎた仕組みは将来むしろ足かせになりうる。モデル更新を「作り直しコスト」ではなく「無料の性能アップ」として取り込める体制（後述の eval 自動化）が競争力になる。

### 学び② 「eval（業務評価セット）」が運用の土台
- **eval（評価セット）とは**：実際の業務タスクと「期待する結果・合格ライン」をセットにし、モデルやエージェントの出力が使い物になるかを**繰り返し自動で測るためのテスト**。いわば「AI 版のユニットテスト」。
- 3社の共通点：
  - **NRI** … 汎用ベンチマークの点数では実務の良し悪しは決まらないとし、①業務文書の解釈 ②帳票・フォーム処理 ③「分からない」と言える正直さ、を**業務タスクで評価**。ダブルチェック業務の置き換えで **50% 効率化**の実績。
  - **楽天** … 自社コードの実 PR を使った独自の評価を構築。コストを「トークン単位」ではなく**「タスク1件あたり」で計測**。
  - 共通の効能：業務単位の eval を一度作れば、**新モデルが出たその日に評価でき、良ければ自動で恩恵を受けられる**。
- **自社への示唆**：EC の価格管理・運用フローに対する「業務 eval セット」を早めに作るほど、今後のモデル更新で自動的に得をする。**最優先の投資対象**。

### 学び③ Claude Code は「人が舵を取る」開発へ
- 新機能は「開発者体験」と「自律化」の2系統（詳細は付録）。要点は、レビュー・並列実行・許可判断・記憶を自動化し、**人は承認待ちの雑務から解放されて方向づけに集中する**こと。
- 印象的だった言葉：仕事の中心が「作業すること」から「**ステアリング（方向づけ）**」へ移る。
- **自社への示唆**：レビューや定例運用の自動化は、価格管理システム改修のような期限の厳しい案件で**人の時間を空ける**効果が大きい。

### 学び④ 本番運用ではコストとコンテキスト管理が勝負（Brad & Rod / Canva）
- **プロンプトキャッシング**：繰り返し使う前置き部分をキャッシュすると、ヒット時は**コストが約1/10**。コンソールにヒット率を見る新ダッシュボードが追加。
- **モデルの使い分け**：常に最上位モデルを使わず、安いモデルを主役に、重要な判断のときだけ上位モデルを「相談役」として呼ぶ（Canva はこれでコストを大幅削減）。
- **自社への示唆**：エージェントを本番運用する段階になったら、ここを押さえないとコストが見合わなくなる。今は「こういう勘所がある」と認識しておけば十分。

### 学び⑤ 全社展開は「そのまま配る」では足りない ― 自社知識の注入（みずほ）
- みずほは内製チーム**40名全員に Claude Code を配布、25プロダクトを並行開発**、社内で100超のエージェントが稼働。
- 鍵は **「ハーネス」＝自社固有の文脈（設計思想・ガバナンス・セキュリティ・開発ルール）を、スキルやプラグインとしてツールに読み込ませる仕組み**。Claude Code をそのまま配っても現場は使いこなせない、という実感ベースの話。
- 思想は **「AI 活用」ではなく「AI 前提の業務プロセス再設計」**。生産性の見方も「何倍速くなったか」ではなく「**何回プロトタイプを作りユーザー検証を回せたか**」へ。
- **自社への示唆**：進行中の社内 Claude Code / Cowork 展開で、自社のドメイン・ルールを「スキル」として整備する設計をそのまま参考にできる。AI リテラシー教育の取り組みとも接続できる。

## 3. すぐにチームへ展開できること（アクション）

前提：担当は **EC サイトのバックエンド開発**。中心スタックは **Ruby on Rails ／ AWS（ECS・Terraform）**。フロントは Vue.js（Vuex）だが実装比重は小さいため軽めに扱う。優先度の高い順（難易度は着手のしやすさ＝低いほどすぐ着手可）。

| # | アクション | 具体的な内容（Rails / AWS 中心） | 期待効果 | 難易度 | 担当/時期 |
|---|---|---|---|---|---|
| 1 | **コードレビューの自動化** | `/ultra-review` を手元の Rails PR で試す → 良ければ GitHub アプリで対象 PR に設定。レビュー観点として **N+1 クエリ・Strong Parameters / 不正な mass assignment・マイグレーションの無停止安全性・Sidekiq ジョブの冪等性**（Rails 側）を与える（バックエンド中心。Vue の PR が来たら状態管理・XSS も観点に追加） | レビュー時間短縮・重大バグの早期検出 | 低 | （記入） |
| 2 | **業務 eval セットの整備** | 価格管理ロジックの代表ケース（**クーポン併用・税計算・会員ランク別価格・送料計算**など）を「入力＋期待する計算結果＋合格ライン」で定義。**既存の RSpec をそのまま eval の土台**にし、「Claude が生成・修正したコードがこの RSpec を通るか」で判定 | モデル更新時に即評価・自動で恩恵／改修の品質担保 | 中 | （記入） |
| 3 | **定例作業の自動化（Routines）** | 「毎朝 **brakeman（Rails 静的セキュリティ）＋ bundler-audit（依存脆弱性）** を走らせ結果を Slack/Discord に要約」「マージ済み PR からリリースノート自動生成」「**CloudWatch アラート**の要約」。既存の AI ニュース収集の仕組みと統合 | セキュリティ雑務の自動化・見落とし防止 | 低 | （記入） |
| 4 | **並列開発（Worktrees）** | 価格管理システム改修の **API 実装（6月末）と ロジックのリファクタ（8月末）を `-w` で別ブランチ並行**。レビュー待ちの間に別タスクを進める | 進行中案件のリードタイム短縮 | 低 | （記入） |
| 5 | **テスト駆動でのコード生成** | みずほの `/TDD` 的に「**先に RSpec を書かせ、最小実装で通す**」フローを自社スラッシュコマンド or `CLAUDE.md` の手順として定義 | 仕様の取り違え防止・手戻り削減 | 中 | （記入） |
| 6 | **`CLAUDE.md` / 設定の棚卸し** | リポジトリの `CLAUDE.md` にある古い指示・不要な作り込みを整理（新モデルは指示追従が強く、古い指定が誤作動しうる） | Fable 5 など新モデルでの不具合予防 | 中 | （記入） |
| 7 | **自社スキル/プラグインの整備（ハーネス）** ← 中核 | 下記「3.1」のスキルをドキュメント化し **Claude Code プラグイン**として配布。**RuboCop / brakeman** など既存の静的解析を併用（みずほの静的ガードレールに相当） | 現場での実用性・品質の底上げ（みずほ型） | 高 | （記入） |

### 3.1 ハーネス（自社スキル）の構成案 ― Rails / AWS 中心

「Claude Code をそのまま配っても現場は使いこなせない」（みずほ）への対策として、自社文脈を注入する。バックエンド担当なので Rails 層と AWS 層を厚く、フロントは軽く。

- **Rails 層（最重要）**：service object / フォームオブジェクト等の自社パターン、API シリアライズ方針（jbuilder / serializer）、命名・ディレクトリ規約、禁止事項（直接 SQL・N+1 を生む書き方など）。
- **AWS / インフラ層**：**ECS（タスク定義・サービス構成）と Terraform の自社流儀**、RDS / ElastiCache 等の構成、**PII・決済まわりのガバナンス**（EC 特有。決済情報の扱いは要注意）。楽天の教訓どおり、**Terraform で IaC 化しておくとエージェントがインフラを読み書きしやすくなる**。
- **フロント（Vue / Vuex）**：実装比重が小さいので深追いしない。必要になったら Vuex のストア構成と自社コンポーネント規約を最小限ドキュメント化する程度でよい。

### 3.2 「まず2週間で」着手するなら

最短で効果が見える組み合わせ：**#1（`/ultra-review` を進行中の価格管理 PR で試す）＋ #2 の第一歩（価格計算ロジックの eval を1本、既存 RSpec ベースで作る）**。どちらも低コストで、いま動いている案件に直接効く。ここで手応えを掴んでから #7 のハーネス整備に広げるのが現実的。

> 注：`/ultra-review` や `-w` などスラッシュ始まりは Claude Code のコマンド。自社でも `CLAUDE.md` や独自スラッシュコマンドで開発フローを型化する発想は参考になる。

## 4. 中期で検討すべきこと

- **Managed Agents / Dynamic Workflows の本格導入**：大規模マイグレーションや夜間の自動監視・修正など、エージェントに丸ごと任せる領域。eval と運用基盤が整ってから。
- **非エンジニアへの展開**：楽天「ガリレオ」、Zapier（800人規模で97%超が毎日 AI 活用）、Epic など、ボトムアップ＋経営層の率先利用が効く。社内リテラシー教育と接続。
- **生産性指標の再設計**：「速度」ではなく「不確実性の低減・検証回数・価値貢献」で語ると社内合意を得やすい（みずほ）。

## 6. 用語集（あいまいになりがちな言葉の定義）

| 用語 | 意味（このカンファレンス文脈での定義） |
|---|---|
| **eval（評価セット）** | 業務タスクと「期待結果・合格ライン」をセットにし、モデル/エージェントの出力が使えるかを繰り返し測るテスト。AI 版のユニットテスト。 |
| **ハーネス（harness）** | モデルに「自社の文脈・ツール・権限」を与えて、実際に仕事をさせるための“受け皿”の仕組み。ルールや知識をスキルとして注入する部分を指すことが多い。 |
| **スキル（skill）** | 業務知識・手順・判断基準などをまとめ、モデルが必要時に参照できる形にしたもの。プロンプトに全部書く代わりに使う。 |
| **MCP** | Model Context Protocol。外部ツールやデータ（社内システム等）にモデルを安全につなぐ共通の接続規格。 |
| **ルーブリック / Outcome** | 「これを満たせば完了」という合格条件の定義。設定は人、達成までの反復は AI が担う（Managed Agents の考え方）。 |
| **コンパクション（compaction）** | 会話が長くなりコンテキスト上限に近づいたとき、いったん要約して圧縮し、続きを実行する仕組み。 |
| **プロンプトキャッシング** | 毎回同じ前置きを送り直す無駄を省き、キャッシュ再利用でコスト・遅延を下げる仕組み（ヒット時コスト約1/10）。 |
| **トークンバジェット** | 1タスクに使ってよいトークン量の上限設定。コストと品質のバランス調整に使う。 |
| **Worktree** | 同一リポジトリで、ブランチを分けたまま複数の作業を並行させる Git の仕組み。Claude Code が裏側を自動処理。 |
| **Routines** | スケジュールや Webhook 等をトリガーに、Claude Code のタスクを定期自動実行する機能。 |
| **Dynamic Workflows** | 大規模なタスクを Claude が自動で分解し、複数エージェントを並列で動かして仕上げる機能（"Ultra Code" がトリガー）。 |
| **Managed Agents** | 自律エージェントの運用に必要な「権限・文脈・インフラ」を Anthropic 側がまとめて提供する新プラットフォーム。 |
| **Human in / on the Loop** | in＝人が逐一確認しながら進める段階／on＝人は監督に回り、必要時だけ介入する段階。AI ネイティブへの移行の中間像。 |
| **スキャフォールディング** | モデルの周辺に置く足場（システムプロンプトや補助コード）。古くなると新モデルで誤作動の原因になりうる。 |

## 付録：セッション別ハイライト（参照用）

主にメインステージ視聴分。詳細は社内 Notion「code with claude 2026」の各メモ参照。

- **Opening keynote / Fable 5・Mythos 5 発表** … 第5世代モデル発表。「次世代前提で設計せよ」。API 流通量 前年比約17倍。
- **Claude Code キーノート（新機能・新インターフェース）** … Desktop GUI、CLI エージェントビュー、コードレビュー、Routines、Dynamic Workflows。社内コードの約80%を Claude が記述。
- **Claude Code 東京：新機能プレゼン** … リモート操作（`/remotecontrol`＋QR）、フルスクリーン/フリッカーフリー、ボイス（`/voice`）、Auto Mode（安全判定して自動実行）、Worktrees（`-w`）、Auto Memory（`MEMORY.md`）、Agent Views。デモは OSS ホワイトボード Excalibur のリポで実演。
- **Claude Managed Agents（製品プレゼン×2）** … 権限・文脈・インフラを一括提供。Outcome ベースの自律反復、Memory、Dreaming、Secret Token Vaults。Notion / Asana が採用。
- **Canva AI 2.0 構築の知見** … ハーネスは作り直し前提（過去3年で3回）、eval は10件から数百件へ拡大、モデル使い分けでコスト80〜90%削減。
- **楽天：AI 活用戦略** … 独自 SWE ベンチ、タスク単位コスト計測、メモリでクリティカルエラー97%減・コスト30%超減、非エンジニア展開「ガリレオ」、リリース四半期→約2週間。「成果の上限はモデルでなくワークフロー・組織設計」。
- **みずほ FG：AI トランスフォーメーション** … 「AI 前提の業務プロセス再設計」、40名全員に Claude Code・25プロダクト並行、多層スキルで自社文脈注入、独自スラッシュコマンド、自社 LLM（オープンウェイトベース）、営業支援 RM Studio（生産性2倍目標）。
- **Theo（capability curve）** … SWE-bench Verified が Sonnet 3.7 約60%→Opus 4.8 で88%。新モデルの3つの改善＝事前計画・失敗からの立て直し（ドゥームループ回避）・長距離タスク保持。開発者向け推奨：eval を作り更新する／古い指示を整理する。
- **NRI：エンタープライズ向けモデル選定** … 汎用ベンチでなく業務タスクで評価（文書解釈・帳票処理・不確実性表明）。業務単位 eval、知識はスキルとして整備。ダブルチェック置換で50%効率化。
- **Brad & Rod：Getting more out of the Claude Platform** … プロンプトキャッシング（ヒット率ダッシュボード）、ツール検索・プログラム的ツール呼び出し・コンパクション、アドバイザー戦略、フォールバック等。
- **辻氏：ソフトウェア民主化・コミュニティ** … 障壁消失、価値は「課題の言語化＋ドメイン知識」へ。Build with Opus ハッカソンは上位5組中4組が非エンジニア。Zapier/Epic 事例。日本アンバサダープログラム募集開始。
